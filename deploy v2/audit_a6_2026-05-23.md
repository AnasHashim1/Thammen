## 1. Headline findings (one paragraph)

A6 is not one bug — it is **a 5× multiplier of a 4s constant**. Every request
pays a ~4s sequential GIS baseline. Slow-path classifications (`standalone_villa`,
`raw_land`, `compound_small`) then pile additional **serial** GIS calls on top:
+21s for villa/land paths, **+101s** for the `compound_small` extent-expansion
(51/835/17 in particular). All Heroku router timeouts (HTTP 503 at 30s) we see
are this extent-expansion explosion. The fix is mechanical, not algorithmic:
**parallelize obviously-independent GIS calls** + **time-budget the BFS
extent-expansion**. Both are isolated, additive, and reversible.

## 3. Per-call latency is essentially fixed at ~830 ms

**Across all 10 GIS layers, every single call lands in 800–900 ms.** Min seen
across 21 runs: 803 ms (`gis.geometry_project`). Typical: ~830 ms. Max: 1 515 ms
on a polygon-heavy `gis.cadastre` call. **One outlier**: an 11 927 ms
(11.9 s) `gis.landuse` call in the very first `a6_trigger_51` rep — likely a
POST-fallback for a many-vertex polygon (Rule #48) plus transient GIS slowness.

The 830 ms floor is **Heroku → GIS Qatar Doha** network round-trip + ArcGIS
server time. We do not control either. **All optimization must reduce call
count or parallelize independent calls** — making any single call faster is
not on the table.

## 6. What is independent vs what is serial (parallelization potential)

Annotated against the actual code paths:

### 6.1 The 4-call lite baseline (`evaluate_unified.py:2478-2517`)
Sequential because each step needs the previous result:
```
qars_primary (find_property: address → PIN, lat/lon)
  → cadastre (get_plot: PIN → polygon)
  → geometry_project (project polygon for shape analysis)
  → districts (district at lat/lon)
```
**Parallelizable subset:** once we have PIN + lat/lon (after `qars_primary`),
the next three are independent of each other:
- `cadastre` needs PIN
- `districts` needs lat/lon
- `geometry_project` needs the polygon from cadastre — so it's chained on
  cadastre, but `districts` can run concurrently with the cadastre+geometry
  pair.

**Savings:** ~800 ms per request (the `districts` call ridden along with
cadastre). **Every request gets this**, including the 4 fast-path cases.

### 6.2 `property_factors.analyze_property` calls (6 independent layers)
`property_factors.py:467` calls all of these against the same `(lat, lon)`:
- `gis.commercial_streets`
- `gis.landmarks`
- `gis.main_roads`
- `gis.local_roads`
- `gis.zoning`
- `gis.landuse`

**All six are independent.** They are completely separate ArcGIS layers, all
keyed on the same point. Currently serial → 6 × 830 ms = **~5 s sequential**.
With `concurrent.futures.ThreadPoolExecutor(max_workers=6)` → **~830 ms +
~50 ms overhead = ~880 ms**.

**Savings:** ~4.1 s per villa/land/compound_small request. Applies to all 4
slow-path cases in the cohort (3 of 7 cohort + the compound_small 4th).

### 6.3 `_expand_extent` BFS neighbour fetching (`qatar_gis.py:1413-1428`)
For each candidate neighbour PIN, calls `self.get_plot(cand['pin'])` which
fires one `cadastre` + one internal `geometry_project`. With 21 neighbours →
42 + 42 = 84 calls, all **serial** in a `while frontier:` loop.

**Two ways to fix:**
- **(a) Time-budget**: hard-cap the loop at e.g. 5 s wall-clock; truncate
  expansion and flag in `extent.notes`. For 51/835/17 this turns ~84 calls
  → maybe 6, capping the call at ~5 s instead of ~70 s. The resulting
  `total_area_m2` is incomplete, but for valuation it just falls back to
  seed plot area (same behaviour as `compound_large` already today).
- **(b) Parallel BFS**: fetch all eligible candidates in parallel up-front
  before the boundary-sharing test. ~21 calls × 830 ms in parallel =
  ~1.2 s instead of ~35 s. Result is *exact*, not truncated.

Both work; (b) is the better answer if `concurrent.futures` is the only new
dependency. (a) is the cheaper if we need to ship today.

### 6.4 What is genuinely serial (cannot be parallelized)
- `qars_primary` → must come first (returns the PIN that all subsequent calls
  need).
- The Asset-Type Reality Check sequence (`qars-in-polygon` + `landuse_at`) is
  data-dependent on the polygon and only runs for the land path — but it's
  only 2 calls.
- The lite-path classification result drives whether the full pipeline runs at
  all. Can't parallelize what we don't know we need.

## 7. Decision-gate report (STOP — do not code yet)

### 7.1 Single largest bottleneck

**`_expand_extent` serial BFS for `compound_small` classifications.** It is
responsible for 71 s of 105 s on 51/835/17 (68 % of total). It is the *only*
phase that scales unbounded with parcel count — every other phase is constant
or bounded ≤3 calls. Without fixing this, **Stage 1 compliance (≤5 s) is
mathematically impossible** for the affected address class.

### 7.2 Top 3 optimization candidates ranked by **impact × effort**

| # | Candidate | Impact (P95) | Code effort | Risk | Stage-1 unlock? |
|---|---|---|---|---|---|
| **A** | **Parallelize `property_factors` 6-layer point-lookup** (ThreadPoolExecutor) | −4.1 s on every slow-path request (villa/land/compound) | **~30 lines, single file**, `property_factors.analyze_property` | **Low** — pure point lookups, no shared state | Brings villa/land from 27 → 23 s. Not Stage 1 alone, but contributes. |
| **B** | **Time-budget `_expand_extent` BFS** (deadline + truncate) | −60 to −95 s on `compound_small` (the 51/835/17 class). All other paths untouched. | **~10 lines** in `qatar_gis._expand_extent`. | **Medium** — truncated `total_area_m2` may understate compound area; affects MUC. Mitigation: flag in `extent.notes` so downstream knows. | Brings 51/835/17 from 100 s → ~5–8 s. **The only path to Stage 1 for this class.** |
| **C** | **Parallelize lite-baseline `districts` with the `cadastre`+`geometry_project` pair** | −0.8 s on **every** request, including the 4 s fast-path cases (which become ~3.3 s) | ~20 lines in `evaluate_unified.py` lite-path block | **Low** — `districts` and `cadastre` are independent once PIN+lat/lon are known | Brings fast-path from 4.1 → 3.3 s. Combined with A: villa/land from 27 → 22 s. |

**Alternative for B** (instead of time-budget): parallel BFS neighbour fetching
using `ThreadPoolExecutor`. Same impact, exact result, ~25 lines, slightly
higher risk (concurrency in a previously-serial loop). Reasonable swap if you
prefer correctness over hard ceilings.

### 7.4 My recommendation (for Anas to approve / reject / amend)

**Phase 2 scope = A + B only**, both single-purpose per Rule #38:

- **2.18.0 = A** (parallelize `property_factors.analyze_property`). One file,
  ~30 lines, low risk, kills 14 % of villa/land latency. Ship first.
- **2.18.1 = B** (time-budget `_expand_extent`). One function, ~10 lines, fixes
  the only existing HTTP 503 class in production. Ship second.

**Defer C** to a follow-up. It's a *constant* −800 ms — nice, but not Stage-1
critical and crosses module boundaries (lite-path is in `evaluate_unified.py`,
not `qatar_gis.py`).

**Defer the "lite + full path GIS deduplication" question** to Sprint 2.18.2
candidate after we have logs from 2.18.0 + 2.18.1 production. It's the only
path to Stage 1 for villa/land (~5 s target), but it's structural — needs its
own audit.

**Out of scope** for this Sprint:
- BUA-aware sanity check (was bundled with A6 in the backlog; orthogonal —
  Rule #38)
- The 11.9 s `gis.landuse` outlier — single observation, n=1; revisit if it
  repeats in production logs after deploy.
- Caching neighbour PIN polygons across requests — adds state, harder to
  reason about for a Stage-1 contract.
